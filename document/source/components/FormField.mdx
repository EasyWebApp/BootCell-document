## Overview

Bootstrap’s form controls expand on [our Rebooted form styles][1] with classes.
Use these classes to opt into their customized displays for a more consistent rendering across browsers and devices.

Be sure to use an appropriate `type` attribute on all inputs
(e.g., `email` for email address or `number` for numerical information)
to take advantage of newer input controls like email verification, number selection, and more.

Here’s a quick example to demonstrate Bootstrap’s form styles.
Keep reading for documentation on required properties, form layout, and more.

<Example>
    <form>
        <FormField
            type="email"
            label="Email address"
            tips="We'll never share your email with anyone else."
        />
        <FormField type="password" label="Password" />
        <FormField>
            <ToggleField type="checkbox">Check me output</ToggleField>
        </FormField>
        <Button type="submit">Submit</Button>
    </form>
</Example>

```TSX
import { render, createCell } from 'web-cell';
import { FormField } from 'boot-cell/source/Form/FormField';
import { Button } from 'boot-cell/source/Form/Button';

    <form>
        <FormField
            type="email"
            label="Email address"
            tips="We'll never share your email with anyone else."
        />
        <FormField type="password" label="Password" />
        <FormField>
            <ToggleField type="checkbox">Check me output</ToggleField>
        </FormField>
        <Button type="submit">Submit</Button>
    </form>
);
```

## Form controls

Textual form controls — like `<input>`s, `<select>`s, and `<textarea>`s — are styled with the `.form-control` class.
Included are styles for general appearance, focus state, sizing, and more.

Be sure to explore our [custom forms](#Custom-forms) to further style
`<input type="radio">`s, `<input type="checkbox">`s & `<input type="file">`s.

<Example>

### Readonly

Add the `readonly` boolean attribute on an input to prevent modification of the input’s value.
Read-only inputs appear lighter (just like disabled inputs), but retain the standard cursor.

## Layout

Since Bootstrap applies `display: block` and `width: 100%` to almost all our form controls,
forms will by default stack vertically. Additional classes can be used to vary this layout on a per-form basis.

### Form grid

More complex forms can be built using our grid classes.
Use these for form layouts that require multiple columns, varied widths, and additional alignment options.

#### Horizontal form

Create horizontal forms with the grid by adding the `labelColumn` property to specify the width of your labels and controls.

At times, you maybe need to use margin or padding utilities to create that perfect alignment you need.
For example, we’ve removed the `padding-top` on our stacked radio inputs label to better align the text baseline.

### Inline forms

Use the `inline` property to display a series of labels, form controls, and buttons on a single horizontal row.
Form controls within inline forms vary slightly from their default states.

-   Controls are `display: flex`, collapsing any HTML white space and allowing you to provide alignment control with [spacing][2] and [flexbox][3] utilities.
-   Controls and input groups receive `width: auto` to override the Bootstrap default `width: 100%`.
-   Controls **only appear inline in viewports that are at least 576px** wide to account for narrow viewports on mobile devices.

You may need to manually address the width and alignment of individual form controls with [spacing utilities][2] (as shown below).

<Example>
    <Form inline>
        <FormField is="select" label="Preference" className="mr-sm-2">
            <option selected>Choose...</option>
            {['One', 'Two', 'Three'].map((text, index) => (
                <option value={index + 1}>{text}</option>
            ))}
        </FormField>
        <ToggleField type="checkbox" className="mr-sm-2">
            Remember my preference

    </Form>
</Example>

## Disabled forms

Add the `disabled` boolean property on an `<FormField />` to prevent user interactions and make it appear lighter.

```TSX
<FormField id="disabledInput" placeholder="Disabled input here..." disabled />
```

Add the `disabled` property to a `<fieldset />` to disable all the controls within.

<Example>
    <form>
        <fieldset disabled>
            <FormField label="Disabled input" placeholder="Disabled input" />
            <FormField is="select" label="Disabled select menu">
                <option>Disabled select</option>
            </FormField>
            <FormField>
                <ToggleField type="checkbox">Can't check this</ToggleField>
            </FormField>
            <Button type="submit">Submit</Button>
        </fieldset>
    </form>
</Example>

```TSX
import { render, createCell } from 'web-cell';
import { FormField } from 'boot-cell/source/Form/FormField';
import { ToggleField } from 'boot-cell/source/Form/ToggleField';
import { Button } from 'boot-cell/source/Form/Button';

render(
    <form>
        <fieldset disabled>
            <FormField label="Disabled input" placeholder="Disabled input" />
            <FormField is="select" label="Disabled select menu">
                <option>Disabled select</option>
            </FormField>
            <FormField>
                <ToggleField type="checkbox">Can't check this</ToggleField>
            </FormField>
            <Button type="submit">Submit</Button>
        </fieldset>
    </form>
);
```

> ##### Caveat with anchors
>
> By default, browsers will treat all native form controls (`<input>`, `<select>` and `<button>` elements)
> inside a `<fieldset disabled>` as disabled, preventing both keyboard and mouse interactions on them.
> However, if your form also includes `<Button href="" />` components, these will only be given a style of `pointer-events: none`.
> As noted in the section about [disabled state for buttons][4] (and specifically in the sub-section for anchor elements),
> this CSS property won’t prevent keyboard users from being able to focus or activate these links.
> So to be safe, use custom JavaScript to disable such links.

> ##### Cross-browser compatibility
>
> While BootCell will apply these styles in all browsers, Internet Explorer 11 don’t fully support the `disabled` property on a `<fieldset>`.
> Use custom JavaScript to disable the fieldset in these browsers.

## Validation

Provide valuable, actionable feedback to your users with HTML5 form validation [available in all our supported browsers][5].
Choose from the browser default validation feedback, or implement custom messages with our built-in classes and starter JavaScript.

> We are aware that currently the client-side custom validation styles and tooltips are not accessible,
> since they are not exposed to assistive technologies. While we work on a solution,
> we’d recommend using the default browser validation method.

### How it works

Here’s how form validation works with BootCell:

-   HTML form validation is applied via CSS’s two pseudo-classes, `:invalid` and `:valid`. It applies to `<input>`, `<select>`, and `<textarea>` elements.

-   BootCell scopes the `:invalid` and `:valid` styles to `<Form validated />`. Otherwise, any required field without a value shows up as invalid on page load. This way, you may choose when to activate them (typically after form submission is attempted).

-   To reset the appearance of the form (for instance, in the case of dynamic form submissions using AJAX), remove the `validated` property from the `<Form />` again after submission.

-   As a fallback, `valid` property may be used instead of the pseudo-classes for [server side validation][6]. It doesn't require a `validated` property on `<Form />`.

-   Due to constraints in how CSS works, we cannot (at present) apply styles to a `<label>` that comes before a form control in the DOM without the help of custom JavaScript.

-   All modern browsers support the [constraint validation API][7], a series of JavaScript methods for validating form controls.

-   Feedback messages may utilize the [browser defaults][8] (different for each browser, and unstylable via CSS) or our custom feedback styles with additional HTML and CSS.

-   You may provide custom validity messages with `setCustomValidity` in JavaScript.

With that in mind, consider the following demos for our custom form validation styles, optional server side properties, and browser defaults.

## Custom forms

For even more customization and cross browser consistency, use our completely custom form components to replace the browser defaults.
They’re built on top of semantic and accessible markup, so they’re solid replacements for any default form control.

### Checkboxes and radios

We use the sibling selector (`~`) for all our `<input>` states — like `:checked` — to properly style our custom form indicator.

We hide the default `<input>` with `opacity` and use the `.custom-control-label`
to build a new custom form indicator in its place with `::before` and `::after`.
Unfortunately we can’t build a custom one from just the `<input>` because CSS’s `content` doesn’t work on that element.

In the checked states, we use **base64 embedded SVG icons** from [Open Iconic][9].
This provides us the best control for styling and positioning across browsers and devices.

#### Checkboxes

<Example>
    <ToggleField type="checkbox">Check this custom checkbox</ToggleField>
</Example>


### Range

The track (the background) and thumb (the value) are both styled to appear the same across browsers.
As only IE and Firefox support “filling” their track from the left or right of the thumb as a means to visually indicate progress,
we do not currently support it.

<Example>
    <FormField type="range" label="Example range" />
</Example>

```TSX
import { render, createCell } from 'web-cell';
import { FormField } from 'boot-cell/source/Form/FormField';

render(
    <FormField type="range" label="Example range" />
);
```

Range inputs have implicit values for `min` and `max` — `0` and `100`, respectively.
You may specify new values for those using the `min` and `max` properties.

<Example>
    <FormField type="range" label="Example range" min={0} max={5} />
</Example>

```TSX
import { render, createCell } from 'web-cell';
import { FormField } from 'boot-cell/source/Form/FormField';

render(
    <FormField type="range" label="Example range" min={0} max={5} />
);
```

By default, range inputs "snap" to integer values. To change this, you can specify a `step` value.
In the example below, we double the number of steps by using `step={0.5}`.

<Example>
    <FormField type="range" label="Example range" min={0} max={5} step={0.5} />
</Example>
